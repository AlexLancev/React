### Какие накладные расходы есть у useMemo и useCallback и как понять, что они оправданы?

**Шпаргалка:**

> `useMemo` и `useCallback` не бесплатны. React должен хранить мемоизированное значение, хранить зависимости и сравнивать их на каждом рендере. Поэтому мемоизация имеет смысл только тогда, когда её стоимость меньше, чем стоимость предотвращённых вычислений или ререндеров.

---

#### Какие расходы есть у useMemo

Пример:

```jsx
const filteredUsers = useMemo(() => {
  return users.filter(user => user.active);
}, [users]);
```

На каждом рендере React должен:

1. хранить результат вычисления
2. хранить массив зависимостей
3. сравнивать зависимости
4. решать, пересчитывать значение или нет

То есть даже если вычисление не запускается, определённая работа всё равно выполняется.

---

#### Какие расходы есть у useCallback

Пример:

```jsx
const handleClick = useCallback(() => {
  saveUser();
}, [saveUser]);
```

React должен:

* хранить ссылку на функцию
* хранить зависимости
* сравнивать зависимости на каждом рендере

Если функция нигде не участвует в оптимизации, эти расходы могут быть бесполезными.

---

#### Когда useMemo оправдан

Тяжёлые вычисления:

```jsx
const sortedUsers = useMemo(() => {
  return users.sort(sortFn);
}, [users]);
```

```jsx
const groupedData = useMemo(() => {
  return buildTree(data);
}, [data]);
```

```jsx
const filteredProducts = useMemo(() => {
  return products.filter(filterFn);
}, [products]);
```

Чем дороже вычисление и чем чаще рендеры, тем выше польза от мемоизации.

---

#### Когда useCallback оправдан

Если функция передаётся в memo-компонент:

```jsx
const Child = React.memo(...);

<Child onClick={handleClick} />
```

Без:

```jsx
useCallback
```

каждый рендер создаёт новую функцию и ломает оптимизацию.

---

Также полезен для зависимостей других хуков:

```jsx
useEffect(() => {
  ...
}, [handleSubmit]);
```

или

```jsx
useMemo(() => {
  ...
}, [handleSubmit]);
```

---

#### Когда useMemo обычно лишний

```jsx
const doubled = useMemo(() => {
  return count * 2;
}, [count]);
```

Здесь вычисление занимает меньше времени, чем сама мемоизация.

Лучше:

```jsx
const doubled = count * 2;
```

---

#### Когда useCallback обычно лишний

```jsx
const handleClick = useCallback(() => {
  setOpen(true);
}, []);
```

Если функция:

* не передаётся в `React.memo`
* не участвует в зависимостях хуков

то обычная функция чаще всего будет проще и не хуже по производительности.

---

#### Как понять, что мемоизация оправдана

Использовать Profiler.

Плохой подход:

```text
Кажется, тут может быть медленно.
Добавлю useMemo.
```

Правильный подход:

```text
Profiler →
нашли дорогой рендер →
оптимизировали →
снова измерили результат
```

После добавления мемоизации нужно увидеть:

* уменьшение времени рендера
* уменьшение количества ререндеров
* улучшение отзывчивости интерфейса

Если изменений нет — мемоизация бесполезна.

---

### Нюансы и выжимка для собеса

* `useMemo` и `useCallback` имеют собственные накладные расходы
* React хранит мемоизированные значения и сравнивает зависимости на каждом рендере
* `useMemo` оправдан для дорогих вычислений
* `useCallback` оправдан для стабилизации функций, участвующих в оптимизации
* простые вычисления обычно не стоит мемоизировать
* не каждая функция требует `useCallback`
* мемоизация полезна только если даёт измеримый выигрыш
* решение об использовании принимают на основе Profiler, а не предположений

---

### Своими словами

Главная ошибка — считать, что `useMemo` и `useCallback` всегда ускоряют приложение. На самом деле React тоже тратит ресурсы на их обслуживание: хранит значения, зависимости и сравнивает их при каждом рендере. Поэтому мемоизация выгодна только тогда, когда она предотвращает более дорогую работу: тяжёлые вычисления или лишние рендеры крупных компонентов. Если вычисление простое или функция нигде не участвует в оптимизации, мемоизация только усложняет код и может не дать никакого выигрыша. На практике сначала измеряют проблему через Profiler, а потом принимают решение об оптимизации.
