### Как вы диагностируете лишние ререндеры конкретного компонента через Profiler и why-did-you-render?

**Шпаргалка:**

Обычно используют два инструмента:

* **React DevTools Profiler** — показывает какие компоненты рендерятся и сколько времени это занимает.
* **why-did-you-render (WDYR)** — показывает почему компонент ререндерился и какие props изменились.


### Нюансы и выжимка для собеса

* Profiler показывает где происходят рендеры и сколько они стоят
* через Flamegraph и Ranked удобно находить самые дорогие компоненты
* Profiler помогает понять какой компонент обновляется слишком часто
* why-did-you-render показывает конкретную причину рендера
* WDYR особенно полезен для `React.memo` компонентов
* частые причины лишних рендеров:

  * новые объекты
  * новые массивы
  * новые функции
  * Context updates
  * ререндер родителя
* после оптимизации нужно повторно измерять результат через Profiler

---

### Своими словами

Profiler отвечает на вопрос **«что рендерится и сколько это стоит?»**, а why-did-you-render отвечает на вопрос **«почему именно это рендерится?»**. Обычно сначала через Profiler находят подозрительный компонент, который часто или долго рендерится. Затем подключают why-did-you-render и смотрят какие props изменяются между рендерами. Очень часто оказывается, что проблема не в данных, а в новых объектах, массивах или функциях, которые создаются на каждом рендере родителя. Это позволяет находить реальные причины лишних обновлений вместо догадок.

#### Диагностика через React DevTools Profiler

##### Шаг 1. Записать сценарий

Открыть:

```text
React DevTools → Profiler
```

Нажать:

```text
Start profiling
```

Выполнить действие:

* ввод в поле
* клик по кнопке
* открытие модалки
* обновление данных

После этого:

```text
Stop profiling
```

---

##### Шаг 2. Найти часто рендерящийся компонент

Вкладка:

```text
Flamegraph
```

или

```text
Ranked
```

Проверяем:

* сколько раз рендерился компонент
* сколько времени занял рендер
* какие компоненты попали в цепочку обновлений

---

##### Шаг 3. Проверить причину рендера

Выбираем компонент.

Profiler покажет:

```text
Why did this render?
```

Например:

```text
Props changed
```

или

```text
Hook changed
```

или

```text
Parent rendered
```

Это первый сигнал, где искать проблему.

#### Диагностика через why-did-you-render

Установка:

```bash
npm install @welldone-software/why-did-you-render
```

Подключение:

```jsx
import React from 'react';

if (process.env.NODE_ENV === 'development') {
  const whyDidYouRender =
    require('@welldone-software/why-did-you-render');

  whyDidYouRender(React, {
    trackAllPureComponents: true
  });
}
```

---

#### Включение для компонента

```jsx
const UserCard = React.memo(function UserCard(props) {
  return <div>{props.name}</div>;
});

UserCard.whyDidYouRender = true;
```

---

#### Что показывает WDYR

Допустим есть:

```jsx
<UserCard
  user={{ name: 'Alex' }}
/>
```

В консоли можно увидеть:

```text
UserCard re-rendered because:

user prop changed

before:
{ name: 'Alex' }

after:
{ name: 'Alex' }
```

На вид данные одинаковые.

Но ссылки разные:

```jsx
{} !== {}
```

Поэтому произошёл ререндер.

---

#### Типичные причины лишних ререндеров

Новые объекты:

```jsx
<Child config={{ theme: 'dark' }} />
```

Новые массивы:

```jsx
<Child items={[1, 2, 3]} />
```

Новые функции:

```jsx
<Child onClick={() => {}} />
```

Изменение Context:

```jsx
useContext(...)
```

Ререндер родителя:

```jsx
Parent render
↓
Child render
```

---

#### Практический процесс диагностики

Обычно последовательность такая:

```text
1. Замечаем проблему
2. Открываем Profiler
3. Находим компонент
4. Смотрим количество рендеров
5. Проверяем причину обновления
6. Подключаем why-did-you-render
7. Смотрим какой prop реально меняется
8. Исправляем проблему
9. Повторно измеряем
```
