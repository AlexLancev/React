### Как нестабильные зависимости в useEffect / useMemo могут приводить к деградации производительности?

**Шпаргалка:**

> React сравнивает зависимости в `useEffect`, `useMemo` и `useCallback` по ссылке. Если в зависимости попадают новые объекты, массивы или функции, React считает их изменившимися и повторно запускает эффект или вычисление, даже если данные визуально не изменились.

---

#### Пример с объектом в useEffect

Плохой пример:

```jsx
function App() {
  const options = {
    page: 1
  };

  useEffect(() => {
    fetchData(options);
  }, [options]);
}
```

На каждом рендере создаётся новый объект:

```jsx
{} !== {}
```

Поэтому эффект будет выполняться снова:

```text
render
↓
new options
↓
useEffect
↓
render
↓
new options
↓
useEffect
...
```

---

#### Решение

```jsx
const options = useMemo(() => {
  return {
    page: 1
  };
}, []);
```

Теперь ссылка стабильна.

---

#### Пример с useMemo

Плохой пример:

```jsx
const filters = {
  active: true
};

const users = useMemo(() => {
  return filterUsers(data, filters);
}, [filters]);
```

Каждый рендер:

```jsx
filters !== previousFilters
```

Поэтому:

```jsx
filterUsers(...)
```

будет выполняться снова.

Фактически мемоизация перестаёт работать.

---

#### Пример с функцией в зависимостях

```jsx
const loadData = () => {
  ...
};

useEffect(() => {
  loadData();
}, [loadData]);
```

На каждом рендере создаётся новая функция:

```jsx
oldFn !== newFn
```

Следовательно эффект запускается снова.

---

#### Реальная проблема — бесконечные запросы

```jsx
function App() {
  const loadUsers = () => {
    fetch('/users');
  };

  useEffect(() => {
    loadUsers();
  }, [loadUsers]);
}
```

Получаем:

```text
render
↓
new loadUsers
↓
effect
↓
request
↓
render
↓
new loadUsers
↓
effect
...
```

В некоторых сценариях это может привести к циклическим обновлениям.

---

#### Частый кейс с React.memo

```jsx
const config = {
  theme: 'dark'
};

<Child config={config} />
```

```jsx
const Child = React.memo(...)
```

Каждый рендер родителя создаёт новый объект.

Для React:

```jsx
oldConfig !== newConfig
```

Следовательно:

```text
React.memo не работает
↓
Child re-render
```

---

#### Как React сравнивает зависимости

Для примитивов:

```jsx
1 === 1
'abc' === 'abc'
true === true
```

Проблем обычно нет.

Для ссылочных типов:

```jsx
{} !== {}
[] !== []
() => {} !== () => {}
```

Даже если содержимое одинаковое.

---

#### Как избежать нестабильных зависимостей

Стабилизировать объекты:

```jsx
const config = useMemo(() => ({
  theme: 'dark'
}), []);
```

Стабилизировать функции:

```jsx
const handleSubmit = useCallback(() => {
  ...
}, []);
```

Зависеть от примитивов:

```jsx
useEffect(() => {
  ...
}, [userId]);
```

Вместо:

```jsx
useEffect(() => {
  ...
}, [user]);
```

если нужен только `user.id`.

---

### Нюансы и выжимка для собеса

* React сравнивает зависимости по ссылке, а не по содержимому
* новые объекты, массивы и функции считаются изменившимися значениями
* нестабильные зависимости могут ломать `useMemo`
* могут вызывать лишние запуски `useEffect`
* могут приводить к лишним запросам, вычислениям и ререндерам
* часто ломают оптимизации через `React.memo`
* для стабилизации используют `useMemo` и `useCallback`
* по возможности в зависимости передают примитивы, а не объекты

---

### Своими словами

Одна из самых частых проблем производительности в React — это нестабильные ссылки. Разработчик может думать, что объект или функция не изменились, но при каждом рендере создаётся новый экземпляр, и для React это уже новое значение. Из-за этого повторно запускаются эффекты, пересчитываются мемоизированные значения и ломается `React.memo`. В результате приложение делает лишнюю работу: дополнительные запросы, вычисления и ререндеры. Поэтому при работе с зависимостями важно понимать разницу между одинаковыми данными и одинаковой ссылкой на объект.
