### В каких случаях React.memo не дает выигрыша или вообще не срабатывает?

**Шпаргалка:**

> `React.memo` работает только если props компонента проходят поверхностное сравнение (`shallow comparison`). Если при каждом рендере приходят новые ссылки на объекты, массивы или функции, мемоизация не поможет. Кроме того, сам `React.memo` имеет накладные расходы и не всегда оправдан.

---

#### Случай №1 — передаются новые объекты

```jsx
<Child options={{ theme: 'dark' }} />
```

На каждом рендере создаётся новый объект:

```jsx
{} !== {}
```

Для React props изменились.

Поэтому:

```jsx
const Child = React.memo(...)
```

не предотвратит ререндер.

---

#### Случай №2 — передаются новые массивы

```jsx
<Child items={[1, 2, 3]} />
```

Каждый рендер:

```jsx
[] !== []
```

React видит новый prop и выполняет рендер.

Решение:

```jsx
const items = useMemo(() => [1, 2, 3], []);
```

---

#### Случай №3 — передаются новые функции

```jsx
<Child onClick={() => console.log('click')} />
```

Каждый рендер создаёт новую функцию:

```jsx
oldFn !== newFn
```

`React.memo` не поможет.

Решение:

```jsx
const handleClick = useCallback(() => {
  console.log('click');
}, []);
```

---

#### Случай №4 — компонент использует Context

```jsx
const theme = useContext(ThemeContext);
```

Даже если props не изменились:

```jsx
<MemoChild />
```

изменение значения Context вызовет новый рендер.

`React.memo` отслеживает только props, но не Context.

---

#### Случай №5 — внутри компонента есть state

```jsx
const Child = React.memo(() => {
  const [count, setCount] = useState(0);

  ...
});
```

При изменении собственного state:

```jsx
setCount(c => c + 1);
```

компонент всё равно рендерится.

`React.memo` защищает только от ререндеров из-за родителя.

---

#### Случай №6 — компонент очень лёгкий

```jsx
const Child = React.memo(() => {
  return <span>Hello</span>;
});
```

Сравнение props тоже требует времени.

Для очень простых компонентов стоимость проверки может быть сопоставима со стоимостью самого рендера.

В таком случае выигрыша может не быть.

---

#### Случай №7 — props действительно часто меняются

```jsx
<Chart data={data} />
```

Если:

```jsx
data
```

обновляется на каждом рендере, то мемоизация бесполезна.

React всё равно будет видеть новые props и выполнять рендер.

---

#### Случай №8 — проблема не в рендерах

Иногда разработчик видит медленный интерфейс и сразу добавляет:

```jsx
React.memo
```

Но причина может быть в другом:

* тяжёлые вычисления
* медленные запросы
* большие списки
* дорогие эффекты
* лишние DOM-операции

В таком случае `React.memo` проблему не решит.

---

### Нюансы и выжимка для собеса

* `React.memo` делает поверхностное сравнение props
* не помогает, если приходят новые объекты, массивы или функции
* не защищает от изменений Context
* не защищает от изменений собственного state компонента
* бесполезен, если props реально меняются на каждом рендере
* может не дать выигрыша для очень простых компонентов
* работает эффективнее в связке с `useMemo` и `useCallback`
* сначала искать проблему через Profiler, потом добавлять мемоизацию

---

### Своими словами

Многие считают, что достаточно обернуть компонент в `React.memo`, и проблема производительности решена. На практике это работает только если props остаются одинаковыми между рендерами. Если родитель каждый раз создаёт новые объекты, массивы или функции, то для React props будут считаться изменившимися и компонент всё равно перерендерится. Также `React.memo` не влияет на изменения state и Context внутри компонента. Поэтому использовать его имеет смысл только после измерений и понимания причины лишних рендеров.
