### Как работает reconciliation в React и как key влияет на алгоритм сопоставления элементов?

**Шпаргалка:**

> Reconciliation — это процесс сравнения предыдущего и нового Virtual DOM. React определяет, что изменилось, и обновляет только необходимые части реального DOM. Для списков React использует `key`, чтобы правильно сопоставлять элементы между рендерами.

---

#### Что происходит при обновлении state

Когда меняется state:

```jsx
setCount(count + 1);
```

React:

1. запускает рендер компонента
2. создаёт новый Virtual DOM
3. сравнивает его с предыдущим Virtual DOM
4. вычисляет минимальный набор изменений
5. обновляет только нужные DOM-узлы

Этот процесс называется **Reconciliation**.

---

#### Пример без изменений

Было:

```jsx
<h1>Hello</h1>
```

Стало:

```jsx
<h1>Hello</h1>
```

React видит, что ничего не изменилось и DOM не трогает.

---

#### Пример с изменением текста

Было:

```jsx
<h1>Hello</h1>
```

Стало:

```jsx
<h1>Hello Alex</h1>
```

React обновит только текст внутри узла:

```html
<h1>Hello Alex</h1>
```

Сам элемент пересоздаваться не будет.

---

#### Как React сравнивает элементы

React использует несколько простых правил.

Если тип элемента изменился:

Было:

```jsx
<div />
```

Стало:

```jsx
<span />
```

React размонтирует старый элемент и создаст новый.

---

Если тип одинаковый:

Было:

```jsx
<button disabled={false} />
```

Стало:

```jsx
<button disabled={true} />
```

React обновит только изменившийся атрибут.

---

#### Проблема списков

Было:

```jsx
['A', 'B', 'C']
```

Стало:

```jsx
['D', 'A', 'B', 'C']
```

Без ключей React сравнивает элементы по позиции:

```text
A → D
B → A
C → B
```

Получается много лишней работы.

---

#### Как помогает key

Было:

```jsx
[
  { id: 1, text: 'A' },
  { id: 2, text: 'B' },
  { id: 3, text: 'C' }
]
```

Стало:

```jsx
[
  { id: 4, text: 'D' },
  { id: 1, text: 'A' },
  { id: 2, text: 'B' },
  { id: 3, text: 'C' }
]
```

React смотрит на ключи:

```text
1 существует
2 существует
3 существует
4 новый
```

и понимает, что нужно добавить только новый элемент.

---

#### Почему index ломает reconciliation

Плохой вариант:

```jsx
items.map((item, index) => (
  <Row key={index} />
))
```

Было:

```text
0 A
1 B
2 C
```

После удаления первого элемента:

```text
0 B
1 C
```

React считает:

```text
A → B
B → C
```

Компоненты начинают переиспользоваться неправильно.

Возможные последствия:

* потеря локального state
* прыгающий фокус
* неправильные данные в input
* лишние ререндеры

---

#### Что происходит со state компонента

React связывает локальный state компонента с его позицией и key.

Если key изменился:

```jsx
<UserForm key="1" />
```

↓

```jsx
<UserForm key="2" />
```

React считает компонент новым:

```text
Unmount старого
Mount нового
```

Весь локальный state будет сброшен.

---

#### Практический пример

Иногда key используют специально:

```jsx
<Form key={userId} />
```

При смене пользователя форма полностью пересоздаётся и очищает своё состояние.

---

### Нюансы и выжимка для собеса

* Reconciliation — алгоритм сравнения старого и нового Virtual DOM
* React обновляет только изменившиеся части DOM
* элементы одного типа переиспользуются, разных типов — пересоздаются
* для списков React использует `key` для сопоставления элементов между рендерами
* `key` должен быть уникальным и стабильным
* хороший `key` уменьшает количество операций обновления DOM
* плохой `key` может приводить к потере state и багам интерфейса
* `index` подходит только для полностью статических списков
* изменение `key` заставляет React размонтировать старый компонент и создать новый

---

### Своими словами

Reconciliation — это механизм, который позволяет React не перерисовывать всю страницу после каждого изменения. React строит новый Virtual DOM, сравнивает его со старым и вычисляет минимальный набор изменений. В списках ключевую роль играет `key`: по нему React понимает, какой элемент остался тем же самым, а какой был добавлен или удалён. Если ключи стабильные, React корректно сохраняет состояние компонентов и обновляет только нужные части интерфейса. Если ключи плохие, например используется индекс в динамическом списке, могут появляться баги и лишние обновления.
