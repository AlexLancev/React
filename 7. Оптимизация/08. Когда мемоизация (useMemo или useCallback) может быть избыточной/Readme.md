### Когда мемоизация (useMemo / useCallback) может быть избыточной?

**Шпаргалка:**

> `useMemo` и `useCallback` — это инструменты оптимизации. Они не ускоряют приложение автоматически. Если использовать их без необходимости, можно получить более сложный код и даже небольшое ухудшение производительности.

---

#### Случай №1 — вычисление слишком дешёвое

Плохой пример:

```jsx
const doubled = useMemo(() => {
  return count * 2;
}, [count]);
```

Здесь вычисление занимает доли микросекунды.

Без мемоизации:

```jsx
const doubled = count * 2;
```

будет проще и зачастую быстрее.

---

#### Случай №2 — дочерний компонент не использует React.memo

```jsx
function Child({ onClick }) {
  return <button onClick={onClick}>Click</button>;
}
```

```jsx
const handleClick = useCallback(() => {
  console.log('click');
}, []);
```

Здесь `useCallback` не даёт выгоды.

Почему?

Потому что `Child` всё равно ререндерится вместе с родителем.

---

#### Случай №3 — рендеры происходят редко

```jsx
function SettingsPage() {
  const [tab, setTab] = useState('general');
}
```

Если компонент обновляется раз в несколько секунд или только по действиям пользователя, оптимизация обычно не нужна.

---

#### Случай №4 — объект никуда не передаётся

```jsx
const options = useMemo(() => {
  return {
    theme: 'dark'
  };
}, []);
```

Если объект используется только внутри компонента:

```jsx
console.log(options.theme);
```

мемоизация бессмысленна.

---

#### Случай №5 — преждевременная оптимизация

Иногда встречается такой код:

```jsx
const value1 = useMemo(...);
const value2 = useMemo(...);
const value3 = useMemo(...);

const fn1 = useCallback(...);
const fn2 = useCallback(...);
const fn3 = useCallback(...);
```

Хотя никаких проблем с производительностью нет.

В результате:

* код становится сложнее
* тяжелее поддерживать
* сложнее искать баги

---

#### Почему useMemo и useCallback тоже имеют стоимость

React должен:

1. хранить мемоизированное значение
2. хранить зависимости
3. сравнивать зависимости на каждом рендере

То есть мемоизация — это не бесплатная операция.

---

#### Когда использовать useMemo

Хорошие кейсы:

```jsx
const sortedUsers = useMemo(() => {
  return users.sort(sortFn);
}, [users]);
```

```jsx
const filteredProducts = useMemo(() => {
  return products.filter(filterFn);
}, [products]);
```

Когда вычисление реально дорогое.

---

#### Когда использовать useCallback

Хорошие кейсы:

```jsx
const handleClick = useCallback(() => {
  ...
}, []);
```

если функция:

* передаётся в `React.memo` компонент
* участвует в зависимостях `useEffect`
* участвует в зависимостях `useMemo`

---

### Нюансы и выжимка для собеса

* мемоизация — инструмент оптимизации, а не правило по умолчанию
* `useMemo` используют для дорогих вычислений
* `useCallback` используют для стабилизации ссылок на функции
* мемоизация имеет собственные накладные расходы
* не стоит мемоизировать простые вычисления и все функции подряд
* если нет проблемы производительности — оптимизация может быть лишней
* сначала измеряем через Profiler, потом добавляем мемоизацию
* преждевременная оптимизация часто ухудшает читаемость кода

---

### Своими словами

Многие разработчики после изучения `useMemo` и `useCallback` начинают добавлять их почти везде. Это ошибка. Мемоизация полезна только тогда, когда она реально уменьшает количество вычислений или ререндеров. Если компонент лёгкий, вычисления дешёвые, а дочерние компоненты не используют `React.memo`, то пользы не будет. На собеседовании важно показать понимание, что мемоизация — это не обязательная практика, а инструмент, который применяется после измерений и обнаружения узкого места.
