### Когда стоит выносить тяжелые вычисления за пределы компонента или в Web Worker?

**Шпаргалка:**

> Если вычисления начинают заметно влиять на отзывчивость интерфейса, их стоит либо вынести из компонента, либо выполнять в Web Worker. Главная цель — не блокировать главный поток браузера (main thread), который отвечает за рендеринг и обработку пользовательских действий.

---

#### Сначала — вынести вычисление из рендера

Плохой пример:

```jsx
function UserList({ users }) {
  const result = expensiveCalculation(users);

  return <List data={result} />;
}
```

Каждый рендер вызывает:

```text
render
↓
expensiveCalculation
↓
UI ждёт завершения
```

---

Лучше:

```jsx
const result = useMemo(() => {
  return expensiveCalculation(users);
}, [users]);
```

Если вычисление действительно дорогое и зависит от props/state.

---

#### Если вычисление не зависит от React

Часто его можно вынести в отдельный модуль:

```jsx
// utils/calculateReport.js

export function calculateReport(data) {
  ...
}
```

```jsx
import { calculateReport } from './utils/calculateReport';
```

Это не ускоряет код само по себе, но:

* упрощает тестирование
* уменьшает размер компонента
* делает логику переиспользуемой

---

#### Когда useMemo уже не помогает

Допустим вычисление занимает:

```text
300–1000+ мс
```

Даже если оно запускается редко, пользователь всё равно может увидеть:

* лаг интерфейса
* зависание скролла
* задержку ввода
* подвисание анимаций

Причина:

```text
JavaScript выполняется в main thread
↓
браузер не может рисовать UI
```

---

#### Когда нужен Web Worker

Web Worker запускает код в отдельном потоке.

Основной поток:

```text
UI
DOM
Events
React
```

Worker:

```text
Тяжёлые вычисления
Парсинг
Обработка данных
```

Работают параллельно.

---

#### Типичные кандидаты для Web Worker

Большие сортировки:

```js
100000+ элементов
```

Обработка файлов:

```js
CSV
Excel
PDF
```

Сложные расчёты:

```js
финансовые модели
аналитика
статистика
```

Графы и алгоритмы:

```js
поиск маршрутов
деревья
кластеризация
```

Обработка изображений:

```js
resize
filters
compression
```

---

#### Пример взаимодействия

Главный поток:

```jsx
worker.postMessage(data);
```

Worker:

```js
self.onmessage = (event) => {
  const result = expensiveCalculation(event.data);

  self.postMessage(result);
};
```

Получение результата:

```jsx
worker.onmessage = (event) => {
  setResult(event.data);
};
```

UI остаётся отзывчивым.

---

#### Когда Web Worker — плохая идея

Для лёгких вычислений:

```js
users.filter(...)
users.map(...)
count * 2
```

Стоимость передачи данных между потоками может оказаться больше самой работы.

---

#### Как принять решение

Обычно порядок такой:

```text
1. Найти проблему через Profiler
2. Убедиться, что тормозит вычисление
3. Попробовать мемоизацию
4. Локализовать ререндеры
5. Если main thread всё ещё блокируется —
   использовать Web Worker
```

---

### Нюансы и выжимка для собеса

* тяжёлые вычисления внутри рендера могут блокировать UI
* сначала стоит уменьшить количество запусков через `useMemo`
* бизнес-логику часто выносят в отдельные функции и модули
* Web Worker нужен, когда вычисления начинают блокировать главный поток браузера
* Worker выполняется в отдельном потоке и не мешает React рендерить интерфейс
* подходит для больших вычислений, обработки файлов, аналитики и сложных алгоритмов
* для простых операций использование Worker обычно не оправдано
* решение принимают после измерений через Profiler, а не заранее

---

### Своими словами

Проблема не в том, что вычисление находится внутри React-компонента, а в том, что оно выполняется в главном потоке браузера. Пока JavaScript считает данные, браузер не может нормально обновлять интерфейс. Если задача относительно небольшая, обычно хватает `useMemo` и уменьшения количества ререндеров. Но если вычисления занимают сотни миллисекунд или секунды, интерфейс начинает подвисать независимо от мемоизации. В таких случаях вычисления переносят в Web Worker, чтобы React и UI продолжали работать параллельно с тяжёлой обработкой данных.
