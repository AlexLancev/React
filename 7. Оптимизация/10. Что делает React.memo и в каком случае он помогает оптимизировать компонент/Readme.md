### Что делает React.memo и в каком случае он помогает оптимизировать компонент?

**Шпаргалка:**

`React.memo` — это HOC (Higher-Order Component), который мемоизирует результат рендера функционального компонента. Если props не изменились, React пропускает повторный рендер компонента.

### Нюансы и выжимка для собеса

* `React.memo` мемоизирует результат рендера функционального компонента
* при одинаковых props компонент не ререндерится
* использует shallow comparison props
* помогает уменьшить количество лишних ререндеров
* особенно полезен для тяжёлых компонентов и больших списков
* новые объекты и функции в props ломают мемоизацию
* часто используется вместе с `useMemo` и `useCallback`
* является аналогом `PureComponent` для функциональных компонентов
* это оптимизация, а не инструмент для исправления архитектурных проблем

---

### Своими словами

Когда родитель ререндерится, React обычно заново вызывает всех его детей. `React.memo` позволяет сказать React: «Если props не изменились, этот компонент можно не трогать». Это особенно полезно для тяжёлых компонентов, которые часто оказываются в дереве рендеринга. Но если в props постоянно создаются новые объекты или функции, мемоизация не сработает, поэтому её часто комбинируют с `useMemo` и `useCallback`.


---

#### Пример

```jsx
const UserCard = React.memo(function UserCard({ name }) {
  console.log('UserCard render');

  return <div>{name}</div>;
});
```

```jsx
function Parent() {
  const [count, setCount] = useState(0);

  return (
    <>
      <button onClick={() => setCount(c => c + 1)}>
        {count}
      </button>

      <UserCard name="Alex" />
    </>
  );
}
```

При изменении `count`:

```jsx
setCount(c => c + 1);
```

Родитель ререндерится.

Но `UserCard` ререндериться не будет, потому что его props не изменились.

---

#### Как это работает

По умолчанию React вызывает дочерние компоненты при каждом ререндере родителя.

`React.memo` добавляет проверку:

```jsx
prevProps === nextProps ?
```

Точнее выполняется **shallow comparison** (поверхностное сравнение).

Если props одинаковые:

```jsx
<UserCard name="Alex" />
```

то React использует предыдущий результат рендера и не вызывает компонент повторно.

---

#### Проблема с объектами и функциями

Мемоизация легко ломается:

```jsx
<UserCard user={{ name: 'Alex' }} />
```

Каждый рендер создаёт новый объект.

Для React:

```jsx
{} !== {}
```

Поэтому компонент будет ререндериваться.

То же самое с функциями:

```jsx
<UserCard onClick={() => {}} />
```

Каждый рендер создаёт новую функцию.

В таких случаях используют:

```jsx
useMemo()
useCallback()
```

для стабилизации ссылок.

---

#### Кастомная функция сравнения

Можно указать свою логику проверки:

```jsx
const UserCard = React.memo(
  UserCardComponent,
  (prevProps, nextProps) => {
    return prevProps.id === nextProps.id;
  }
);
```

Если функция возвращает `true`, ререндер пропускается.

---

#### Когда React.memo действительно полезен

Подходит для:

* тяжёлых компонентов
* больших списков
* таблиц
* графиков
* сложных UI-виджетов
* компонентов, которые часто получают одинаковые props

---

#### Когда React.memo бесполезен

```jsx
function SimpleText({ text }) {
  return <span>{text}</span>;
}
```

Если компонент маленький и рендерится быстро, стоимость сравнения props может быть выше потенциальной выгоды.
