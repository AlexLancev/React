### Как “state colocation” помогает уменьшить количество ререндеров?

**Шпаргалка:**

State colocation — это подход, при котором состояние хранится максимально близко к компоненту, который его использует. Чем меньше область видимости state, тем меньше компонентов будут ререндериться при его изменении.

### Нюансы и выжимка для собеса

* state colocation — хранение состояния максимально близко к месту использования
* изменение state вызывает ререндер компонента и его потомков
* чем выше находится state, тем больше компонентов может обновиться
* локальный state обычно дешевле глобального
* не стоит поднимать состояние без необходимости
* состояние поднимают только если его используют несколько компонентов
* чрезмерное использование Context часто нарушает принцип state colocation
* этот подход уменьшает область обновлений и количество лишних ререндеров

---

### Своими словами

State colocation — это один из самых простых и эффективных способов оптимизации React-приложения. Вместо того чтобы хранить состояние в верхней части дерева или в глобальном хранилище, его держат рядом с компонентом, которому оно нужно. Тогда изменение этого состояния затрагивает только небольшой участок интерфейса. Многие проблемы производительности решаются не мемоизацией, а правильным размещением state: чем меньше компонентов зависит от него, тем меньше лишних ререндеров происходит.

---

#### Что такое state colocation

Идея простая:

```text
Хранить state там, где он нужен,
а не поднимать его выше без необходимости.
```

---

#### Проблема поднятого состояния

Плохой пример:

```jsx
function App() {
  const [inputValue, setInputValue] = useState('');

  return (
    <>
      <Header />
      <Sidebar />
      <Input
        value={inputValue}
        onChange={setInputValue}
      />
      <Footer />
    </>
  );
}
```

При каждом:

```text
onChange
↓
setInputValue
↓
App render
↓
Header render
Sidebar render
Input render
Footer render
```

Хотя значение нужно только Input.

---

#### State рядом с местом использования

Лучше:

```jsx
function Input() {
  const [value, setValue] = useState('');

  return (
    <input
      value={value}
      onChange={e => setValue(e.target.value)}
    />
  );
}
```

Теперь:

```text
onChange
↓
Input render
```

Остальное дерево не затрагивается.

---

#### Пример с модальным окном

Частая ошибка:

```jsx
function App() {
  const [isOpen, setIsOpen] = useState(false);

  return (
    <>
      <BigDashboard />
      <Modal />
    </>
  );
}
```

Открытие модалки:

```text
setIsOpen
↓
App render
↓
BigDashboard render
↓
Modal render
```

---

Лучше:

```jsx
function ModalContainer() {
  const [isOpen, setIsOpen] = useState(false);

  return (
    <>
      <OpenButton />
      <Modal />
    </>
  );
}
```

Теперь обновляется только небольшой участок дерева.

---

#### Почему это улучшает производительность

React обновляет компонент, в котором изменился state, и его потомков.

Чем выше расположен state:

```text
больше поддерево
↓
больше потенциальных ререндеров
```

Чем ниже расположен state:

```text
меньше поддерево
↓
меньше работы для React
```

---

#### Когда state действительно нужно поднимать

Если данные используются несколькими компонентами:

```jsx
<SearchInput />
<SearchResults />
```

Оба зависят от:

```jsx
query
```

Тогда логично поднять состояние к общему родителю.

---

#### Связь с Context

Частая ошибка:

```text
Любой state сразу кладём в Context
```

В результате:

```text
локальное состояние
↓
становится глобальным
↓
растёт количество подписчиков
↓
растут ререндеры
```

Сначала стоит спросить:

```text
Кому реально нужен этот state?
```

---

#### Связь с производительностью форм

Плохой вариант:

```jsx
<Form>
  50 полей
</Form>
```

Один общий state:

```jsx
formData
```

Изменение одного поля:

```text
обновляет всю форму
```

---

Лучше:

```jsx
<Field />
<Field />
<Field />
```

Каждое поле хранит локальное состояние или имеет отдельную подписку.
