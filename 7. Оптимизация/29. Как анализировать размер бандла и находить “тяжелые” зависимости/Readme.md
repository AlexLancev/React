### Как анализировать размер бандла и находить "тяжелые" зависимости?

**Шпаргалка:**

> Анализ бандла позволяет понять, какие библиотеки занимают больше всего места в итоговом JavaScript. Цель — найти самые тяжёлые зависимости, дубликаты пакетов и код, который можно вынести в отдельные чанки.

---

#### Зачем анализировать бандл

Симптомы:

```text
Долгая загрузка приложения
Большой main.js
Низкие оценки Lighthouse
Долгий Parse/Compile JavaScript
```

Часто проблема не в React, а в:

```text
Больших npm-пакетах
Лишних зависимостях
Неправильном импортировании библиотек
Отсутствии code splitting
```

---

#### Самый популярный инструмент — Bundle Analyzer

Для Webpack:

```bash
npm install --save-dev webpack-bundle-analyzer
```

После сборки получаем интерактивную карту:

```text
main.js
├─ react
├─ react-dom
├─ lodash
├─ chart.js
├─ moment
└─ ...
```

Размер каждого пакета отображается визуально.

---

#### Для Vite

Обычно используют:

```bash
rollup-plugin-visualizer
```

После сборки открывается отчёт:

```text
dist/stats.html
```

с визуализацией всех чанков.

---

#### Что искать в первую очередь

##### Очень большие библиотеки

Например:

```text
moment
chart.js
monaco-editor
ckeditor
echarts
mapbox-gl
```

Часто именно они занимают большую часть бандла.

---

##### Дубликаты зависимостей

Например:

```text
lodash v4
lodash-es v4
```

или:

```text
date-fns v2
date-fns v3
```

В бандл попадают обе версии.

---

##### Неиспользуемый код

Например:

```js
import _ from 'lodash';
```

вместо:

```js
import debounce from 'lodash/debounce';
```

Можно случайно подтянуть всю библиотеку ради одной функции.

---

#### Проверка через Chrome DevTools Coverage

Открываем:

```text
DevTools
↓
Coverage
```

Получаем:

```text
Загружено: 500 KB
Использовано: 120 KB
```

Если большая часть кода не используется на текущей странице, это кандидат на code splitting.

---

#### Tree Shaking

Современные сборщики умеют удалять неиспользуемый код.

Работает лучше с:

```js
import { debounce } from 'lodash-es';
```

чем с:

```js
const _ = require('lodash');
```

---

#### Анализ чанков

Смотреть:

```text
main.js
vendors.js
route chunks
```

Если:

```text
main.js = 2 MB
```

это повод разобраться, что туда попало.

---

Хороший результат:

```text
main.js
↓
минимальный набор для старта
```

Остальное:

```text
admin.chunk.js
charts.chunk.js
editor.chunk.js
```

загружается по необходимости.

---

#### Что делать после обнаружения тяжёлой зависимости

Варианты:

##### 1. Lazy loading

```jsx
const Editor = React.lazy(() =>
  import('./Editor')
);
```

---

##### 2. Замена библиотеки

Например:

```text
moment → dayjs
```

или:

```text
lodash → точечные импорты
```

---

##### 3. Удаление неиспользуемой зависимости

Иногда пакет остался после старого функционала и больше не используется.

---

##### 4. Динамический импорт

```js
const module = await import('./heavy-module');
```

Код попадёт в отдельный chunk.

---

#### Что смотреть в Lighthouse

Особенно полезны:

```text
Reduce unused JavaScript
Minimize main-thread work
JavaScript execution time
```

Эти метрики часто напрямую связаны с размером бандла.

---

### Нюансы и выжимка для собеса

* размер бандла анализируют через Bundle Analyzer или Visualizer
* цель — найти самые тяжёлые пакеты, дубликаты и неиспользуемый код
* Chrome Coverage показывает, сколько загруженного JS реально используется
* большие зависимости часто выносят в отдельные чанки через lazy loading
* важно использовать tree shaking и точечные импорты
* после анализа обычно применяют code splitting, замену библиотек или удаление лишних зависимостей
* главный вопрос — что попало в initial bundle и можно ли это загружать позже

---

### Своими словами

Когда приложение начинает долго загружаться, первым делом стоит посмотреть не на React, а на размер бандла. Для этого используют Bundle Analyzer или Visualizer, которые показывают, какие библиотеки занимают больше всего места. Часто выясняется, что половину бандла занимает редактор, карта или графическая библиотека, которая нужна только на одной странице. После этого такие зависимости выносят в отдельные чанки через lazy loading или заменяют на более лёгкие аналоги. Основная цель анализа — сделать стартовый bundle как можно меньше и загружать тяжёлый код только тогда, когда он действительно нужен пользователю.
