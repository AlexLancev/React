### Почему дочерний компонент рендерится даже если его props «визуально» не изменились?

**Шпаргалка:**

> По умолчанию React не сравнивает props дочерних компонентов. Если родительский компонент рендерится, React запускает рендер всех его дочерних компонентов независимо от того, изменились props или нет.

---

#### Базовый пример

```jsx
function Child({ name }) {
  console.log('Child render');

  return <div>{name}</div>;
}

function Parent() {
  const [count, setCount] = useState(0);

  return (
    <>
      <button onClick={() => setCount(c => c + 1)}>
        {count}
      </button>

      <Child name="Alex" />
    </>
  );
}
```

При каждом изменении `count`:

```jsx
setCount(c => c + 1);
```

происходит:

```text
Parent render
Child render
```

Хотя:

```jsx
name="Alex"
```

не изменился.

---

#### Почему так происходит

При рендере родителя React заново вызывает его функцию:

```jsx
Parent()
```

Во время выполнения снова создаётся дерево компонентов:

```jsx
<Child name="Alex" />
```

Поэтому React запускает рендер дочернего компонента.

Это нормальное поведение React.

---

#### React.memo меняет ситуацию

```jsx
const Child = React.memo(function Child({ name }) {
  console.log('Child render');

  return <div>{name}</div>;
});
```

Теперь React выполнит поверхностное сравнение props.

Если было:

```jsx
name="Alex"
```

и осталось:

```jsx
name="Alex"
```

то рендер Child будет пропущен.

---

#### Частая причина — новые объекты

Кажется, что props не изменились:

```jsx
<Child options={{ theme: 'dark' }} />
```

Но на самом деле при каждом рендере создаётся новый объект:

```jsx
{} !== {}
```

Для React props изменились.

Поэтому даже с `React.memo` компонент будет рендериться.

---

#### Решение через useMemo

```jsx
const options = useMemo(() => {
  return { theme: 'dark' };
}, []);

<Child options={options} />
```

Теперь ссылка остаётся стабильной.

---

#### Частая причина — новые функции

```jsx
<Child onClick={() => console.log('click')} />
```

Каждый рендер создаёт новую функцию:

```jsx
oldFn !== newFn
```

Для React props изменились.

---

#### Решение через useCallback

```jsx
const handleClick = useCallback(() => {
  console.log('click');
}, []);

<Child onClick={handleClick} />
```

Ссылка на функцию остаётся прежней.

---

#### Важно понимать

Ререндер компонента не означает обновление DOM.

React может:

```text
1. Вызвать функцию компонента
2. Сравнить результат
3. Не изменить DOM вообще
```

Поэтому сам факт рендера ещё не означает проблему производительности.

---

### Нюансы и выжимка для собеса

* по умолчанию рендер родителя вызывает рендер всех дочерних компонентов
* React не делает автоматическое сравнение props обычных компонентов
* `React.memo` добавляет поверхностное сравнение props
* новые объекты и массивы имеют новую ссылку и считаются изменившимися
* новые функции также считаются новыми значениями props
* для стабилизации объектов используют `useMemo`
* для стабилизации функций используют `useCallback`
* ререндер компонента не равен обновлению DOM
* оптимизировать стоит только те рендеры, которые реально влияют на производительность

---

### Своими словами

Многие думают, что если значение props визуально одинаковое, то дочерний компонент не должен рендериться. Но React работает иначе: когда рендерится родитель, он заново строит всё дерево компонентов, поэтому дочерние компоненты тоже вызываются. Чтобы React начал пропускать такие рендеры, нужен `React.memo`. Но даже с ним можно случайно ломать оптимизацию, если передавать новые объекты, массивы или функции на каждом рендере. Поэтому часто `React.memo` используют вместе с `useMemo` и `useCallback`.
