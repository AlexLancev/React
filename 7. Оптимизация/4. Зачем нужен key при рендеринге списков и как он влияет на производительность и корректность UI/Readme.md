### Зачем нужен key при рендеринге списков и как он влияет на производительность и корректность UI?

**Шпаргалка:**

> `key` — это уникальный идентификатор элемента в списке. React использует его во время Reconciliation, чтобы понимать какой элемент был добавлен, удалён, перемещён или изменён.

---

#### Пример

```jsx
const users = [
  { id: 1, name: 'Alex' },
  { id: 2, name: 'John' }
];

return (
  <>
    {users.map(user => (
      <UserCard
        key={user.id}
        user={user}
      />
    ))}
  </>
);
```

Здесь:

```jsx
key={user.id}
```

помогает React однозначно определить каждый элемент списка.

---

#### Как React работает без key

Допустим есть список:

```jsx
['A', 'B', 'C']
```

После обновления:

```jsx
['D', 'A', 'B', 'C']
```

Без ключей React не понимает, что появился новый элемент.

Он может решить:

```jsx
A -> D
B -> A
C -> B
```

и обновить все элементы списка.

---

#### Как React работает с key

Если есть ключи:

```jsx
[
  { id: 1, text: 'A' },
  { id: 2, text: 'B' },
  { id: 3, text: 'C' }
]
```

и появляется:

```jsx
{ id: 4, text: 'D' }
```

React видит:

```jsx
id:1 существует
id:2 существует
id:3 существует
id:4 новый
```

и создаёт только новый элемент.

---

#### Почему key влияет на производительность

Правильные ключи позволяют React:

* выполнять меньше операций сравнения
* переиспользовать существующие DOM-узлы
* избегать лишних размонтирований и монтирований компонентов

Особенно важно для:

* больших списков
* таблиц
* виртуализации
* динамических форм

---

#### Почему key влияет на корректность UI

Рассмотрим список input-полей:

```jsx
{items.map(item => (
  <input
    key={item.id}
    defaultValue={item.value}
  />
))}
```

Если использовать неправильный key, React может перепутать элементы.

Последствия:

* потеря фокуса
* неправильные значения в input
* сброс локального state компонента
* странное поведение анимаций

---

#### Почему нельзя использовать index

Плохой вариант:

```jsx
{items.map((item, index) => (
  <UserCard
    key={index}
    user={item}
  />
))}
```

Если удалить первый элемент:

Было:

```jsx
0 A
1 B
2 C
```

Стало:

```jsx
0 B
1 C
```

Для React:

```jsx
A -> B
B -> C
```

Он начинает переиспользовать не те компоненты.

Из-за этого появляются баги со state и UI.

---

#### Когда index допустим

Можно использовать индекс если список:

* статический
* никогда не сортируется
* никогда не фильтруется
* элементы не удаляются и не добавляются

Например:

```jsx
['React', 'Vue', 'Angular']
```

которые всегда отображаются в одном порядке.

---

#### Какие ключи использовать

Лучший вариант:

```jsx
key={item.id}
```

Также подходят:

```jsx
key={user.uuid}
key={product.sku}
key={message.id}
```

Главное требование:

* уникальность среди соседних элементов
* стабильность между рендерами

---

### Нюансы и выжимка для собеса

* `key` нужен React для корректного сравнения списков при Reconciliation
* ключ должен быть уникальным и стабильным
* правильный `key` помогает переиспользовать DOM-элементы и уменьшать количество операций обновления
* неправильный `key` может приводить к потере state, фокуса и багам в UI
* `index` как `key` — частая ошибка на собеседованиях
* использовать индекс можно только для полностью статических списков
* лучший источник ключа — уникальный id из данных

---

### Своими словами

`key` нужен не для самого JavaScript и не для map, а для React. По ключу React понимает, какой элемент списка соответствует какому элементу после обновления данных. Если ключи хорошие, React обновляет только нужные элементы. Если ключи плохие или используется index в динамическом списке, React может перепутать элементы, из-за чего появляются баги с input, локальным state и производительностью. На практике почти всегда нужно использовать настоящий уникальный id из данных.
