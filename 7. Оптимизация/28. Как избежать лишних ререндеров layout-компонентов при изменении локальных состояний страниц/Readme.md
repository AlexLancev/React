### Как избежать лишних ререндеров layout-компонентов при изменении локальных состояний страниц?

**Шпаргалка:**

> Если состояние относится только к странице, оно должно жить внутри страницы. Чем выше в дереве находится state, тем больше компонентов будут ререндериться при его изменении.

### Нюансы и выжимка для собеса

* хранить state максимально близко к месту использования (state colocation)
* не поднимать локальное состояние страницы в Layout или App без необходимости
* стабильный Layout должен выступать оболочкой для страниц через children или Outlet
* React.memo помогает только если props остаются стабильными
* Context может провоцировать массовые обновления подписчиков
* сначала уменьшать область обновления state, потом применять мемоизацию
* если при изменении страницы рендерятся Header и Sidebar — обычно state расположен слишком высоко

---

### Своими словами

Самый эффективный способ избежать лишних ререндеров Layout-компонентов — не хранить состояние страниц в Layout. Если поиск нужен только на странице поиска, то и state должен жить там же. Часто разработчики поднимают состояние слишком высоко, а потом пытаются лечить проблему через React.memo, useMemo и useCallback. Обычно правильнее сначала локализовать state, чтобы изменение данных затрагивало только нужную часть дерева компонентов. Тогда Layout с Header, Sidebar и навигацией остаётся практически неизменным и не участвует в лишних рендерах.


---

#### Типичная проблема

Допустим есть структура:

```text
App
└─ Layout
   ├─ Header
   ├─ Sidebar
   └─ Page
```

Плохо:

```jsx
function Layout() {
  const [search, setSearch] = useState('');

  return (
    <>
      <Header />
      <Sidebar />
      <Page
        search={search}
        setSearch={setSearch}
      />
    </>
  );
}
```

Теперь изменение:

```text
search
```

вызывает ререндер:

```text
Layout
↓
Header
↓
Sidebar
↓
Page
```

Хотя Header и Sidebar вообще не используют это состояние.

---

#### State Colocation

Лучше хранить состояние максимально близко к месту использования.

```jsx
function Page() {
  const [search, setSearch] = useState('');

  return (
    <>
      <SearchInput
        value={search}
        onChange={setSearch}
      />
    </>
  );
}
```

Теперь изменение search ререндерит только:

```text
Page
↓
SearchInput
```

---

#### Мемоизация Layout-компонентов

Если Layout получает стабильные props:

```jsx
const Header = React.memo(HeaderComponent);
const Sidebar = React.memo(SidebarComponent);
```

React сможет пропускать их рендеры.

---

#### Не хранить page state в App

Плохо:

```jsx
<App>
  state всех страниц
</App>
```

Получается:

```text
изменение любого поля
↓
App render
↓
всё дерево вниз
```

---

Лучше:

```text
каждая страница
↓
свой локальный state
```

---

#### Осторожно с Context

Плохо:

```jsx
<AppContext.Provider
  value={{
    search,
    setSearch
  }}
>
```

При изменении search:

```text
Provider value изменился
↓
все подписчики контекста обновились
```

Даже если им search не нужен.

---

Лучше:

* разделять контексты
* использовать селекторы
* мемоизировать value

---

#### Layout как стабильная оболочка

Хорошая архитектура:

```jsx
<Layout>
  <Outlet />
</Layout>
```

Layout содержит:

```text
Header
Sidebar
Navigation
```

Страница рендерится через:

```text
Outlet
```

Изменение состояния страницы влияет только на содержимое Outlet.

---

#### Когда помогает React.memo

Если Layout получает:

```jsx
<Layout>
  {children}
</Layout>
```

и собственные props не меняются:

```jsx
export default React.memo(Layout);
```

может сократить лишние ререндеры.

Но если каждый раз создаются новые props:

```jsx
<Layout config={{ theme: 'dark' }} />
```

мемоизация не поможет.

---

#### Что смотреть в Profiler

Если при вводе в поле страницы рендерятся:

```text
Header
Sidebar
Navbar
Footer
```

это признак того, что состояние поднято слишком высоко.

Часто решение:

```text
переместить state ниже
```

а не добавлять useMemo и React.memo везде подряд.
