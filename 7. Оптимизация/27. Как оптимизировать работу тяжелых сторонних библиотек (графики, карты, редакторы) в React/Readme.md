### Как оптимизировать работу тяжелых сторонних библиотек (графики, карты, редакторы) в React?

**Шпаргалка:**

> Тяжёлые библиотеки (Charts, Maps, Rich Text Editors) часто становятся узким местом не из-за React, а из-за большого объёма JS-кода, сложного DOM и дорогих операций рендера. Основные способы оптимизации: lazy loading, изоляция ререндеров, мемоизация данных и уменьшение количества обновлений.

### Нюансы и выжимка для собеса

* тяжёлые библиотеки часто оптимизируют через lazy loading и code splitting
* данные и конфиги для графиков желательно мемоизировать
* тяжёлые компоненты стоит изолировать от лишних ререндеров через `React.memo`
* для больших наборов данных используют агрегацию и downsampling
* карты часто оптимизируют через кластеризацию маркеров и ленивую инициализацию
* редакторы нередко работают эффективнее в uncontrolled режиме
* тяжёлую подготовку данных можно выносить в Web Worker
* сначала нужно понять, где узкое место: React-рендеры или сама библиотека

---

### Своими словами

При работе с графиками, картами и редакторами проблема часто не в React, а в том, что сама библиотека тяжёлая. Поэтому первая задача — не загружать её раньше времени через lazy loading. Вторая — не заставлять её лишний раз рендериться. Для этого мемоизируют данные и конфиги, используют React.memo и локализуют состояние. Если библиотека работает с огромным объёмом данных, обычно уменьшают этот объём ещё до рендера, а самые дорогие вычисления выносят в Web Worker. В реальных проектах именно сочетание code splitting, уменьшения количества ререндеров и подготовки данных даёт основной прирост производительности.

---

#### Проблема №1. Большой размер bundle

Часто библиотеки вроде:

* Chart.js
* ECharts
* Leaflet
* Mapbox
* Monaco Editor
* TinyMCE
* CKEditor

весят сотни килобайт или даже мегабайты.

Если импортировать напрямую:

```jsx
import MonacoEditor from '@monaco-editor/react';
```

код попадёт в основной bundle.

---

#### Решение: lazy loading

```jsx
const MonacoEditor = React.lazy(() =>
  import('@monaco-editor/react')
);
```

```jsx
<Suspense fallback={<EditorSkeleton />}>
  <MonacoEditor />
</Suspense>
```

Редактор загрузится только тогда, когда действительно понадобится.

---

#### Проблема №2. Частые ререндеры

Плохо:

```jsx
<Chart data={data} />
```

где:

```jsx
const data = transformData(rawData);
```

создаётся на каждом рендере.

---

Лучше:

```jsx
const data = useMemo(() => {
  return transformData(rawData);
}, [rawData]);
```

Теперь график получает стабильные данные.

---

#### Проблема №3. Пересоздание конфигураций

Плохо:

```jsx
<Chart
  options={{
    animation: true,
    legend: true
  }}
/>
```

На каждом рендере создаётся новый объект.

---

Лучше:

```jsx
const options = useMemo(() => ({
  animation: true,
  legend: true
}), []);
```

---

#### Проблема №4. Ререндер тяжёлого компонента из-за родителя

Например:

```jsx
Dashboard
 ├─ SearchInput
 ├─ Filters
 └─ HeavyChart
```

Изменение поиска:

```text
SearchInput
↓
Dashboard render
↓
HeavyChart render
```

хотя график не изменился.

---

Решение:

```jsx
const HeavyChart = React.memo(HeavyChartComponent);
```

или вынести состояние ближе к месту использования.

---

#### Проблема №5. Большие наборы данных

Например:

```text
100 000 точек на графике
```

Даже идеальный React не спасёт.

Используют:

* агрегацию
* downsampling
* пагинацию
* отображение части данных

---

Вместо:

```text
100000 точек
```

часто отображают:

```text
1000–5000 точек
```

без потери визуального качества.

---

#### Проблема №6. Карты

Карты особенно дорогие:

```text
Mapbox
Google Maps
Leaflet
```

Потому что одновременно работают:

* Canvas/WebGL
* события мыши
* геоданные
* маркеры

---

Полезные техники:

##### Кластеризация маркеров

Вместо:

```text
5000 маркеров
```

рисуем:

```text
20 кластеров
```

---

##### Ленивая инициализация

```jsx
{
  isVisible && <Map />
}
```

или через Intersection Observer.

Карта создаётся только после появления в viewport.

---

#### Проблема №7. Rich Text Editors

Редакторы часто имеют собственное состояние.

Плохо:

```jsx
setValue(...)
↓
React render
↓
Editor render
```

на каждый символ.

---

Часто используют:

* debounce
* uncontrolled режим
* обновление по blur
* внутренний state редактора

---

#### Проблема №8. Дорогие вычисления перед рендером

Например:

```jsx
const series = buildChartData(data);
```

если функция занимает:

```text
100–300 мс
```

нужно:

```jsx
useMemo(...)
```

или переносить вычисления в:

```text
Web Worker
```

---

#### Когда нужен Web Worker

Хорошие кандидаты:

* обработка больших CSV
* аналитика
* статистика
* георасчёты
* подготовка данных для графиков

---

Пример:

```text
серверные данные
↓
Worker
↓
готовые series
↓
Chart
```

UI остаётся отзывчивым.

---

#### Что проверять в Profiler

Если тормозит график или карта:

смотреть:

```text
Render duration
Commit duration
Количество ререндеров
```

Если компонент рендерится часто:

```text
React проблема
```

Если рендерится редко, но долго:

```text
сама библиотека тяжёлая
```

и оптимизация нужна уже на её уровне.
