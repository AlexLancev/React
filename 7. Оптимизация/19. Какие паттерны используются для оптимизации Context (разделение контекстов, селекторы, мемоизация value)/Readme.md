### Какие паттерны используются для оптимизации Context (разделение контекстов, селекторы, мемоизация value)?

**Шпаргалка:**

> Основная идея оптимизации Context — уменьшить количество компонентов, которые получают обновление. Для этого используют разделение контекстов, мемоизацию `value` и селективные подписки (selectors).

---

#### 1. Разделение контекстов (Context Splitting)

Плохой вариант:

```jsx
<AppContext.Provider
  value={{
    user,
    theme,
    cart,
    notifications
  }}
>
```

Любое изменение:

```jsx
cart
```

или

```jsx
notifications
```

обновит всех подписчиков контекста.

---

Лучше:

```jsx
<UserContext.Provider>
  <ThemeContext.Provider>
    <CartContext.Provider>
      ...
    </CartContext.Provider>
  </ThemeContext.Provider>
</UserContext.Provider>
```

Теперь:

```text
изменился cart
↓
обновились только подписчики CartContext
```

---

#### 2. Мемоизация value

Плохой вариант:

```jsx
<ThemeContext.Provider
  value={{
    theme,
    toggleTheme
  }}
>
```

На каждом рендере создаётся новый объект:

```jsx
{} !== {}
```

React считает:

```text
value изменился
↓
обновить подписчиков
```

---

Правильно:

```jsx
const value = useMemo(() => ({
  theme,
  toggleTheme
}), [theme, toggleTheme]);

<ThemeContext.Provider value={value}>
```

Теперь объект стабилен между рендерами.

---

#### 3. Стабилизация функций через useCallback

Плохой вариант:

```jsx
const toggleTheme = () => {
  setTheme(t => !t);
};
```

Каждый рендер создаёт новую функцию.

---

Лучше:

```jsx
const toggleTheme = useCallback(() => {
  setTheme(t => !t);
}, []);
```

Теперь ссылка на функцию не меняется.

---

#### 4. Разделение State и Actions

Частый паттерн в крупных приложениях:

```jsx
<UserStateContext.Provider value={user}>
<UserActionsContext.Provider value={actions}>
```

где:

```jsx
const actions = {
  login,
  logout,
  updateProfile
};
```

Компоненты, которым нужны только действия:

```jsx
const { logout } = useContext(UserActionsContext);
```

не будут обновляться при изменении данных пользователя.

---

#### 5. Context Selectors

Проблема обычного Context:

```jsx
const state = useContext(AppContext);
```

Подписка идёт на весь объект.

Даже если нужен только:

```jsx
state.user.name
```

обновление любого поля вызывает ререндер.

---

Решение — селекторы:

```jsx
const userName = useContextSelector(
  AppContext,
  state => state.user.name
);
```

Теперь компонент реагирует только на изменение:

```jsx
state.user.name
```

а не всего состояния.

---

#### 6. Использование store с селективными подписками

Например:

```jsx
const user = useStore(state => state.user);
```

или

```jsx
const theme = useStore(state => state.theme);
```

Подписка происходит только на выбранный фрагмент состояния.

Это одна из причин популярности Zustand, Redux Toolkit и других state-менеджеров.

---

#### Когда какой подход использовать

| Подход                  | Что решает                                     |
| ----------------------- | ---------------------------------------------- |
| Разделение контекстов   | уменьшает область обновлений                   |
| useMemo для value       | убирает лишние обновления из-за новых объектов |
| useCallback для actions | стабилизирует функции                          |
| State / Actions Context | уменьшает число подписчиков                    |
| Context Selectors       | обновляет только нужные компоненты             |
| Store с селекторами     | масштабируется лучше обычного Context          |

---

### Нюансы и выжимка для собеса

* главная проблема Context — обновление всех подписчиков при изменении value
* первый уровень оптимизации — разделять контексты по ответственности
* объект `value` обычно мемоизируют через `useMemo`
* функции внутри `value` стабилизируют через `useCallback`
* часто разделяют данные и действия на разные контексты
* селекторы позволяют подписываться только на нужный кусок состояния
* для больших приложений селективные подписки работают эффективнее обычного Context
* Context хорошо подходит для редко меняющихся данных, но не всегда оптимален как глобальный state manager

---

### Своими словами

Большинство проблем с Context возникает не из-за самого API, а из-за того, что в один контекст складывают слишком много данных. Тогда любое изменение начинает "будить" большое количество компонентов. Поэтому оптимизация обычно сводится к уменьшению области обновления: разделить контексты, не создавать новые объекты и функции без необходимости и подписывать компоненты только на те данные, которые им действительно нужны. Чем точнее подписка, тем меньше лишних ререндеров.
