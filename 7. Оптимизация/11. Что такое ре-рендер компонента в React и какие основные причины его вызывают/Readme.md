**Шпаргалка:**

Ре-рендер (re-render) — это повторный вызов функции компонента для вычисления нового Virtual DOM. После этого React сравнивает старое и новое дерево (Reconciliation) и обновляет реальный DOM только там, где есть изменения.

### Нюансы и выжимка для собеса

* ре-рендер = повторный вызов функции компонента
* основные причины:

  * изменение state
  * изменение props
  * ре-рендер родителя
  * изменение Context
* изменение переменной или `useRef.current` ререндер не вызывает
* ререндер ≠ изменение DOM
* после рендера React выполняет Reconciliation и обновляет только изменившиеся части DOM
* большинство проблем производительности связаны не с самим ререндером, а с тяжёлой логикой внутри него или слишком большим количеством лишних ререндеров

---

### Своими словами

Когда что-то меняется в React (state, props или context), React заново запускает функцию компонента и пересчитывает JSX. Это и есть ре-рендер. При этом браузер не обязательно будет что-то перерисовывать на экране. Сначала React сравнивает новый результат со старым через Virtual DOM и обновляет только реальные изменения. Поэтому сам по себе ре-рендер — это нормально. Оптимизировать нужно не количество ререндеров любой ценой, а случаи, когда они приводят к заметным затратам по времени или вызывают лишние вычисления.

---

#### Основные причины ре-рендера

**1. Изменение state**

```jsx
const [count, setCount] = useState(0);

setCount(1);
```

Самая частая причина.

Любой вызов `setState` (`useState`, `useReducer`) запускает повторный рендер компонента.

---

**2. Изменение props**

```jsx
<Child name={name} />
```

Если родитель передал новые props, дочерний компонент ререндерится.

---

**3. Ре-рендер родительского компонента**

```jsx
function Parent() {
  const [count, setCount] = useState(0);

  return <Child />;
}
```

При ререндере `Parent` React по умолчанию заново вызывает и `Child`, даже если его props не изменились.

---

**4. Изменение Context**

```jsx
const value = useContext(UserContext);
```

Если изменилось значение провайдера, все подписанные потребители будут ререндерены.

---

**5. Принудительный ререндер**

Редкий случай.

```jsx
setState(prev => prev);
```

или

```jsx
forceUpdate()
```

(в классовых компонентах).

---

#### Что НЕ вызывает ре-рендер

Изменение обычной переменной:

```jsx
let count = 0;

count++;
```

Изменение значения в `ref`:

```jsx
const ref = useRef();

ref.current = 123;
```

React не отслеживает такие изменения.

---

#### Как это работает

При ререндере:

1. React вызывает компонент заново.
2. Создаётся новый Virtual DOM.
3. React сравнивает его с предыдущим.
4. В реальный DOM попадают только необходимые изменения.

Важно понимать: **ререндер компонента ≠ обновление DOM**.

Компонент может ререндериться, но DOM останется без изменений.
