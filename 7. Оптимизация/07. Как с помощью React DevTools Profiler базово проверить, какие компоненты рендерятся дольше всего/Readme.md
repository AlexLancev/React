### Как с помощью React DevTools Profiler базово проверить, какие компоненты рендерятся дольше всего?

**Шпаргалка:**

React DevTools Profiler — это инструмент для анализа производительности React-приложения. Он показывает какие компоненты рендерились, сколько времени занял каждый рендер и какие компоненты являются узкими местами.

### Нюансы и выжимка для собеса

* Profiler — главный инструмент анализа производительности React
* позволяет увидеть время рендера каждого компонента
* Flamegraph показывает структуру дерева и стоимость рендеров
* Ranked помогает быстро найти самые тяжёлые компоненты
* смотреть нужно на:

  * Actual Duration
  * количество рендеров
  * компоненты с наибольшим временем выполнения
* сначала измеряем проблему, потом применяем оптимизации
* Profiler помогает понять, где действительно нужны React.memo, useMemo и useCallback

---

### Своими словами

Profiler — это как диспетчер задач для React-компонентов. Он показывает, кто именно тратит время во время рендера. Вместо догадок можно увидеть реальные цифры: какие компоненты рендерятся, сколько раз и сколько миллисекунд на это уходит. Обычно сначала открывают Profiler, воспроизводят проблему, находят самые дорогие компоненты через Flamegraph или Ranked, а уже потом принимают решение об оптимизации. Это правильный подход, потому что позволяет оптимизировать реальные узкие места, а не весь код подряд.

---

#### Шаг 1. Открыть Profiler

Установить React DevTools и открыть вкладку:

```text
React DevTools → Profiler
```

---

#### Шаг 2. Начать запись

Нажать кнопку:

```text
Start profiling
```

После этого выполнить действие в приложении:

* клик по кнопке
* ввод в input
* открытие модального окна
* переключение вкладок

То есть воспроизвести сценарий, который хочется проверить.

---

#### Шаг 3. Остановить запись

Нажать:

```text
Stop profiling
```

React сохранит информацию о всех рендерах, произошедших во время записи.

---

#### Шаг 4. Изучить Flamegraph

Основной режим:

```text
Flamegraph
```

Каждый прямоугольник — отдельный компонент.

Чем шире прямоугольник:

```text
██████████
```

тем больше времени занял его рендер.

Обычно самые широкие элементы — первые кандидаты на оптимизацию.

---

#### Шаг 5. Смотреть время рендера

При выборе компонента Profiler показывает:

```text
Actual duration
```

Фактическое время рендера компонента.

Например:

```text
UserTable
Actual duration: 18ms
```

```text
Button
Actual duration: 0.3ms
```

Очевидно, что оптимизировать кнопку смысла нет.

---

#### Шаг 6. Проверить количество рендеров

Profiler показывает:

```text
Render #1
Render #2
Render #3
```

Если компонент рендерится десятки раз при одном действии пользователя — стоит разобраться почему.

---

#### Режим Ranked

Очень полезный режим:

```text
Ranked
```

Он автоматически сортирует компоненты по времени рендера.

Вместо дерева сразу видно:

```text
UserTable     22ms
Sidebar       14ms
Chart         10ms
Button         0.2ms
```

Так проще находить самые дорогие компоненты.

---

#### Что делать после обнаружения проблемы

Если найден тяжёлый компонент:

Проверить:

* нет ли лишних ререндеров
* нужен ли `React.memo`
* нужен ли `useMemo`
* нужен ли `useCallback`
* не выполняются ли тяжёлые вычисления внутри render
* можно ли разбить компонент на более мелкие

---

#### Что НЕ стоит делать

Неправильно:

```text
Сначала добавить React.memo везде,
а потом смотреть результат.
```

Правильно:

```text
Profiler → найти проблему →
измерить → оптимизировать.
```
