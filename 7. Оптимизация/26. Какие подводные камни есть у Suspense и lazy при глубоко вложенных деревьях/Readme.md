### Какие подводные камни есть у Suspense и lazy при глубоко вложенных деревьях?

**Шпаргалка:**

> `React.lazy` и `Suspense` упрощают code splitting, но при неправильном размещении могут приводить к большим областям загрузки, мерцанию интерфейса и каскадным задержкам. Основная задача — правильно выбрать границы (`Suspense boundaries`).

### Нюансы и выжимка для собеса

* `Suspense` скрывает всё дерево внутри своей границы до завершения загрузки
* слишком высокий `Suspense` может скрывать большие части интерфейса
* лучше делать небольшие локальные Suspense boundaries
* вложенные lazy-компоненты могут создавать waterfall loading
* fallback должен быть максимально похож на конечный UI (skeleton лучше spinner)
* не стоит создавать lazy-компоненты для мелких UI-элементов
* `Suspense` не заменяет `ErrorBoundary`
* границы загрузки нужно проектировать с точки зрения UX, а не только производительности

---

### Своими словами

Главная ошибка при использовании `Suspense` — поставить его слишком высоко в дереве. Тогда загрузка одного небольшого lazy-компонента может скрыть половину страницы. Вторая частая проблема — каскадная загрузка, когда один lazy-компонент начинает загружаться только после загрузки другого. Поэтому на практике стараются делать несколько независимых границ Suspense вокруг крупных блоков интерфейса. Тогда графики, таблицы и другие тяжёлые части страницы загружаются отдельно и не мешают друг другу. Также важно помнить, что Suspense решает только состояние загрузки, а ошибки загрузки чанков должны обрабатываться через Error Boundary.

---

#### Проблема №1. Слишком высокий Suspense

Допустим:

```jsx
<Suspense fallback={<Loader />}>
  <App />
</Suspense>
```

Внутри:

```text
App
├─ Header
├─ Sidebar
├─ Content
└─ LazyChart
```

Если загружается:

```text
LazyChart
```

то React покажет:

```text
Loader
```

вместо всего дерева.

Пользователь временно потеряет:

```text
Header
Sidebar
Content
```

хотя они уже были готовы.

---

Лучше:

```jsx
<>
  <Header />
  <Sidebar />

  <Suspense fallback={<ChartLoader />}>
    <LazyChart />
  </Suspense>
</>
```

Теперь загрузка графика не влияет на остальной UI.

---

#### Проблема №2. Большие области перерисовки

Плохо:

```jsx
<Suspense fallback={<PageLoader />}>
  <Dashboard />
</Suspense>
```

Внутри Dashboard:

```text
Chart
Table
Map
Stats
```

Если один компонент ожидает загрузку:

```text
Map
```

может скрыться весь Dashboard.

---

Обычно лучше делать несколько границ:

```jsx
<Suspense fallback={<ChartLoader />}>
  <Chart />
</Suspense>

<Suspense fallback={<MapLoader />}>
  <Map />
</Suspense>
```

---

#### Проблема №3. Водопад загрузок (waterfall)

Пример:

```jsx
const Page = lazy(() => import('./Page'));
```

После загрузки Page:

```jsx
const Chart = lazy(() => import('./Chart'));
```

После загрузки Chart:

```jsx
const Analytics = lazy(() => import('./Analytics'));
```

Получаем:

```text
Page
↓
Chart
↓
Analytics
```

Каждый chunk ждёт предыдущий.

Это называется:

```text
waterfall loading
```

и увеличивает время загрузки страницы.

---

#### Проблема №4. Мерцание интерфейса

Допустим пользователь уже работает с приложением.

При условном рендере:

```jsx
{
  showChart && <LazyChart />;
}
```

может появиться:

```text
контент
↓
fallback
↓
контент
```

Пользователь видит скачки интерфейса.

Поэтому fallback желательно делать похожим на будущий UI:

```jsx
<SkeletonChart />
```

а не:

```jsx
<Spinner />
```

---

#### Проблема №5. Слишком много маленьких чанков

Иногда начинают делать lazy буквально для всего:

```jsx
LazyButton
LazyAvatar
LazyBadge
```

Получается:

```text
много сетевых запросов
↓
сложнее загрузка
↓
выигрыша нет
```

Code splitting эффективен только для достаточно крупных частей приложения.

---

#### Проблема №6. Повторное появление fallback

Если lazy-компонент размонтируется:

```jsx
{
  isOpen && <LazyModal />;
}
```

то при следующем открытии возможны повторные состояния загрузки (зависит от сценария и кэширования).

Нужно учитывать UX и поведение загрузчиков.

---

#### Проблема №7. Обработка ошибок

`Suspense` обрабатывает ожидание загрузки:

```text
loading state
```

но не ошибки загрузки чанка.

Например:

```text
chunk load failed
network error
```

Для этого нужен:

```jsx
<ErrorBoundary>
```

```jsx
<ErrorBoundary>
  <Suspense fallback={<Loader />}>
    <LazyComponent />
  </Suspense>
</ErrorBoundary>
```

---

#### Хороший паттерн

```jsx
<Layout>
  <Header />

  <Sidebar />

  <Suspense fallback={<ChartSkeleton />}>
    <Chart />
  </Suspense>

  <Suspense fallback={<TableSkeleton />}>
    <Table />
  </Suspense>
</Layout>
```

Каждый тяжёлый блок загружается независимо.
