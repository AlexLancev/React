### Как React-приложение влияет на Core Web Vitals (LCP, CLS, INP) и какие оптимизации это улучшают?

**Шпаргалка:**

> Core Web Vitals — это набор пользовательских метрик производительности. React напрямую не измеряет их, но архитектура приложения, размер бандла, количество ререндеров и стратегия загрузки сильно влияют на итоговые показатели.

### Нюансы и выжимка для собеса

* LCP отвечает за скорость появления основного контента
* LCP улучшают через code splitting, уменьшение bundle, lazy loading и SSR
* CLS показывает визуальную стабильность интерфейса
* CLS улучшают через резервирование места под контент и использование skeleton-заглушек
* INP измеряет отзывчивость интерфейса на действия пользователя
* INP улучшают через уменьшение ререндеров, виртуализацию, мемоизацию и вынос тяжёлых вычислений
* большой JavaScript bundle негативно влияет сразу на несколько Core Web Vitals
* оптимизация React-приложения напрямую отражается на пользовательском опыте и SEO

---

### Своими словами

Core Web Vitals показывают не скорость React, а то, насколько быстро и комфортно пользователь может пользоваться приложением. LCP страдает от больших бандлов и тяжёлой загрузки страницы, CLS — от прыгающего интерфейса, а INP — от медленных реакций на действия пользователя. Поэтому большинство оптимизаций React сводится к трём вещам: уменьшить объём загружаемого кода, не допускать лишних ререндеров и не блокировать главный поток тяжёлыми вычислениями. Если приложение быстро показывает контент, не дёргает верстку и мгновенно реагирует на клики, то Core Web Vitals обычно находятся в хорошей зоне.

## LCP (Largest Contentful Paint)

### Что измеряет

Время до отображения самого крупного видимого элемента страницы:

```text
Hero Image
Большой заголовок
Баннер
Главный контент
```

Чем меньше значение — тем быстрее пользователь видит основной контент.

---

### Как React ухудшает LCP

Большой bundle:

```text
HTML
↓
скачать JS
↓
выполнить JS
↓
отрендерить React
↓
показать контент
```

Пользователь долго ждёт появления страницы.

---

Лишние зависимости:

```text
Monaco
Charts
Maps
Editor
```

могут попасть в initial bundle и замедлить старт.

---

### Как улучшить LCP

#### Code Splitting

```jsx
const AdminPage = lazy(() => import('./AdminPage'));
```

---

#### Lazy Loading тяжёлых компонентов

```jsx
const Chart = lazy(() => import('./Chart'));
```

---

#### Оптимизация изображений

```html
<img
  src="hero.webp"
  width="1200"
  height="600"
/>
```

---

#### SSR / SSG

Сервер сразу отдаёт готовый HTML:

```text
HTML
↓
контент уже виден
↓
гидратация React
```

---

#### Уменьшение размера bundle

Использовать:

```text
tree shaking
code splitting
bundle analysis
```

---

## CLS (Cumulative Layout Shift)

### Что измеряет

Насколько сильно интерфейс "прыгает" во время загрузки.

---

Плохо:

```text
Заголовок
↓
загрузилась реклама
↓
контент сдвинулся вниз
```

---

### Как React ухудшает CLS

Условный рендер:

```jsx
{
  isLoaded && <Banner />;
}
```

После загрузки появляется новый блок и двигает весь интерфейс.

---

Изображения без размеров:

```jsx
<img src="image.jpg" />
```

Браузер не знает размеры заранее.

---

Поздняя загрузка компонентов:

```jsx
<Suspense fallback={null}>
  <HeavyWidget />
</Suspense>
```

После загрузки страница может резко измениться.

---

### Как улучшить CLS

#### Всегда резервировать место

```jsx
<img
  width={800}
  height={400}
/>
```

---

#### Использовать Skeleton

Вместо:

```jsx
fallback={null}
```

лучше:

```jsx
fallback={<ChartSkeleton />}
```

---

#### Фиксированные размеры контейнеров

```css
.chart {
  min-height: 400px;
}
```

---

#### Осторожно с динамическими блоками

Заранее предусматривать место для:

```text
баннеров
виджетов
рекламы
графиков
```

---

## INP (Interaction to Next Paint)

### Что измеряет

Скорость реакции интерфейса на действия пользователя:

```text
Клик
Ввод текста
Открытие меню
Скролл
```

---

### Как React ухудшает INP

Большие ререндеры:

```text
Клик
↓
обновляется большой state
↓
рендерится половина приложения
```

---

Тяжёлые вычисления:

```jsx
const result = expensiveCalculation(data);
```

---

Большие списки:

```text
5000 элементов
```

перерисовываются на каждый ввод.

---

Тяжёлые обработчики событий:

```jsx
onChange
↓
API запрос
↓
фильтрация
↓
сортировка
↓
рендер
```

---

### Как улучшить INP

#### State Colocation

Держать state ближе к месту использования.

---

#### React.memo

```jsx
const List = React.memo(ListComponent);
```

---

#### useMemo

```jsx
const filtered = useMemo(() => {
  return expensiveFilter(data);
}, [data]);
```

---

#### Виртуализация списков

Например:

```text
10000 элементов
↓
отрисовываем 30 видимых
```

---

#### Debounce

Для поиска:

```text
ввод
↓
300 мс
↓
запрос
```

---

#### Web Worker

Тяжёлые вычисления не блокируют главный поток.

---

## Что обычно сильнее всего влияет на CWV в React

Чаще всего проблемы создают:

```text
Большой initial bundle
Тяжёлые библиотеки
Лишние ререндеры
Большие списки
Отсутствие code splitting
Изображения без размеров
Тяжёлые вычисления в UI-потоке
```
